## 왜 OKR을 쓰는가

작은 팀에서도 한 달이 지나면 "지금 우리가 정확히 어디를 향하고 있지?"가 흐려지기 시작합니다.<br>
한 명 한 명은 열심히 일하지만, 그 노력이 같은 방향을 가리키고 있는지는 확신하기 어렵습니다.<br>

OKR은 이 질문에 답하기 위한 도구입니다. 두 가지를 강제합니다:<br>

>1. **무엇을 할 것인가** (Tasks) — 이번 스프린트에 집중할 일<br>
>2. **그게 됐다는 걸 어떻게 알 것인가** (Key Results) — 측정 가능한 결과<br>

**"열심히 했다"가 아니라 "이런 결과가 나왔다"로 말하기 위한 약속**입니다.<br>

## 우리의 구조: Tasks와 Key Results

```
[Tasks]   메시지 프로토콜 안정화
  │
  ├─ [KR] 클라우드 → 서버 메시지 전송 99% 성공률
  │   │
  │   ├─ [sub-issue] MQTT QoS 설정 변경 (firmware repo)
  │   └─ [sub-issue] 재전송 로직 구현 (edge-server repo)
  │
  └─ [KR] 메시지 유실률 1% 미만
      │
      └─ [sub-issue] E2E 통합 테스트 (ml repo)
```

세 가지 레벨이 있습니다:

| 레벨 | 무엇 | 어디에 | 누가 작성 |
|---|---|---|---|
| **Tasks** | 스프린트 단위 큰 작업 묶음 | `dev-okr` repo | 책임자 |
| **Key Results** | 측정 가능한 결과 (2-4개) | `dev-okr` repo | 책임자 |
| **Sub-issues** | 실제 코드 작업 단위 | 각 코드 repo | 작업자 |

**Tasks는 "왜·무엇"을, KR은 "어떻게 검증"을, Sub-issue는 "어떻게 만들지"를 답합니다.**<br>

각각 다른 repo에 존재해도, GitHub의 sub-issue 기능으로 hierarchy view에서 한눈에 보입니다.<br>

## Tasks 작성하기

Task는 스프린트의 큰 단위입니다. 보통 스프린트당 1-3개.

### 좋은 Task의 조건

1. **한 문장으로 설명할 수 있는 명확한 주제**<br>
2. **"왜 지금 이걸 하는가"가 분명함**<br>
3. **2-4개의 KR로 측정 가능함**<br>
4. **스프린트 안에 끝낼 수 있는 규모**<br>

### "Tasks 설명" 작성법

1-2 문단으로, 짧게. 단 **6개월 후 다시 봐도 맥락이 살아있도록** 작성하세요.<br>
약어와 내부 용어는 풀어서 적어주세요.<br>

✅ **좋은 예:**
> 현재 ESP32-S3 펌웨어와 N100 엣지 서버 간 MQTT 통신에서 간헐적 메시지 유실이 관찰되고 있음.<br>
> 이로 인해 음향 데이터가 누락되어 ML 모델 학습 데이터 품질에 영향을 주고 있음.<br>
> 이번 스프린트에서 메시지 프로토콜의 신뢰성을 검증하고 유실률을 1% 미만으로 낮춘다.<br>

❌ **나쁜 예:**
> MQTT 개선

### "Tasks 수행의 목적" 작성법

"왜 지금 이걸 하는가"를 회사·제품·팀 관점에서 답합니다.<br>

작성할 때 자문해보세요:<br>
- 이 Task가 안 되면 무엇이 막히는가?<br>
- 다음 스프린트·다음 단계 계획에 어떻게 연결되는가?<br>
- 고객·사용자에게 어떤 영향이 있는가?<br>

### "예상 결과물" 작성법

스프린트 종료 시 "이게 됐다"고 말할 수 있는 상태를 묘사합니다.<br>
숫자보다 정성적 묘사 OK — 정량 검증은 KR이 합니다.<br>

## Key Results 작성하기

KR은 OKR의 핵심입니다. **잘못 쓰면 OKR 전체가 무의미해집니다.**<br>

### 좋은 KR의 4가지 조건

1. **측정 가능** — 객관적인 숫자 또는 명확한 완료 기준<br>
2. **검증 가능** — 누가 봐도 같은 결론<br>
3. **시간 제약** — 이번 스프린트 안에<br>
4. **결과 중심** — 작업 자체가 아니라 작업이 만드는 *결과*<br>

### KR 작성의 황금 공식

> **[지표]를 [시작값]에서 [목표값]으로**

예시:
- "DCASE Task 2 fan 데이터셋 AUC를 0.67에서 0.85 이상으로"
- "엣지→클라우드 메시지 유실률을 5%에서 1% 미만으로"
- "자체 fan 데이터를 200샘플에서 1,000샘플로"

### "측정 방법"이 가장 중요합니다

스프린트 말에 "달성했다 vs 못 했다"로 다투는 경우의 **99%는 측정 방법이 모호했기 때문**입니다.

다음 3가지를 반드시 명확히:

- **방법**: 어떻게 계산하나? (예: "테스트셋 1,000개 추론 후 AUC 계산")
- **데이터/도구**: 무엇으로 측정? (예: "scikit-learn `roc_auc_score`, 평가 스크립트 `eval_auc.py`")
- **측정 주기**: 언제 측정? (예: "매주 금요일 오후")

KR을 등록하기 전에 이 세 줄이 **구체적인 명사로** 채워졌는지 확인하세요.

### Task vs KR — 흔한 혼동

KR 작성하시다 보면 "이건 작업인지 결과인지" 헷갈리는 순간이 옵니다.

| ❌ Task처럼 쓴 KR | ✅ 결과 중심 KR |
|---|---|
| "MQTT 재전송 로직 구현" | "메시지 유실률 1% 미만 달성" |
| "LSTM-AE 모델 학습 코드 작성" | "AUC 0.85 이상 모델 확보" |
| "데이터 수집 자동화 스크립트 제작" | "정상 700 + 이상 300 샘플 수집" |

**규칙**: 동사가 "구현/작성/개발/제작"으로 끝나면 KR이 아니라 Task의 sub-issue입니다.

## Confidence 사용법

Confidence는 단순한 진척 %보다 훨씬 유용합니다. **"이번 스프린트 안에 정말 될까?"에 대한 솔직한 답**입니다.

| 등급 | 의미 | 권장 행동 |
|---|---|---|
| 🟢 70%+ | 자신 있음. 정상 진행하면 달성. | 계속 진행 |
| 🟡 30-70% | 가능하지만 노력·운 필요. | 위험 요소 면밀히 관찰 |
| 🔴 <30% | Stretch goal. 어려움 인지하고도 도전. | 의식적 자원 투입 또는 재정의 |

### 솔직한 작성이 핵심

- 🟢 일변도는 도전이 부족합니다. **스프린트당 최소 1개는 🟡이나 🔴**이어야 합니다.
- 🔴 일변도는 욕심이 과한 것일 수 있습니다. 모두 안 되면 팀이 지칩니다.
- **체크인할 때마다 갱신하세요.** 시작 시 🟢이었지만 진행 중 🔴이 될 수 있고, 그건 잘못이 아닙니다.

### Confidence가 변할 때 액션

| 변화 | 무엇을 해야 하나 |
|---|---|
| 🟢 → 🟡 | 무엇 때문에? 이슈 코멘트에 한 줄 기록 |
| 🟡 → 🔴 | 도움 요청 시점. `#okr-tracking` 채널에 공유 |
| 🔴 유지 | 미드 스프린트 리뷰에서 "포기·재정의·자원 추가" 결정 |

## 운영 리듬

### 스프린트 정의
스프린트는 공통 목표를 가지고 정해진 기간 동안 집중적으로 일을 수행하는 단위로,<br>
"전력질주(sprint)"에서 따온 말 입니다.<br>
짧은 호흡으로 명확한 결과물을 만드는 데 목적이 있으며, 프로덕트의 추가 기능과 개선 사항을 중심으로 운영할 예정입니다.<br>

### 스프린트 구성

플래닝 — 시작 시점, 목표와 작업 합의<br>
스크럼 — 중간 점검, 진행 공유와 블로커 발굴<br>
회고 — 종료 시점, 과정과 장애 요인 돌아보기<br>

### 스프린트 주기

기본 주기: 2주

### 플래닝

방식: 비동기. GitHub의 Sprint Planning 이슈에서 진행합니다.<br>
참여: 각자 본인 섹션에 capacity, 가져갈 이슈, 우려/블로커, 제안을 기입합니다.<br>
확정: 플래닝 마감 시각에 결정 사항이 잠기며, 이후 변경은 별도 절차로 처리합니다.<br>

### 스크럼

빈도: 매주 월요일, 금요일 두 차례 진행합니다. 데일리는 운영하지 않습니다.<br>

공유 항목:
- 지난 점검 이후 진행한 일<br>
- 다음 점검까지 할 일<br>
- 막힌 부분, 도움이 필요한 부분<br>
**길게 끌지 않습니다. 추가 논의가 필요한 주제는 별도 채널로 분리합니다.**

### 회고

시점: 스프린트 종료일<br>
목적: 결과를 기반으로 과정·방식·장애 요인을 점검하고, 다음 스프린트를 더 낫게 만듭니다.<br>
다루는 것: 무엇이 막혔는가, 왜 막혔는가, 다음에 어떻게 풀 것인가.<br>
다루지 않는 것: 개인에 대한 비난, 결과만 보는 평가.<br>

### 그라운드 룰

1. 회고는 반성의 시간이 아닙니다. 더 나아가기 위한 생각의 시간입니다.<br>
2. 사람이 아니라 프로세스를 봅니다.<br>
3. 딜레이가 있었다면 "누구의 잘못인지"가 아니라 "무엇이 그것을 야기했는지"에 초점을 둡니다.<br>
4. 어떤 형태의 비난·비판도 허용되지 않습니다.<br>

## 자주 하는 실수

### 1. 작업 자체를 KR로 쓴다
- ❌ "API 엔드포인트 5개 구현"
- ✅ "p95 응답시간 200ms 이하 달성"

### 2. 측정 방법을 비워둔다
스프린트 말 분쟁의 99%가 여기서 옵니다. "구체적으로 어떻게 잴 거냐"는 질문에 한 줄로 답할 수 없으면 측정 방법이 부족합니다.

### 3. Confidence를 안 갱신한다
작성하고 한 번도 안 바꾸면 죽은 신호입니다. 매주 체크인마다 한 번씩은 갱신해주세요.

### 4. KR이 너무 많다
Task당 KR 2-4개가 적당합니다. 5개 넘으면 우선순위가 없는 것과 같습니다.

### 5. 🟢만 잡는다
모두 달성 가능한 안전한 목표만 잡으면 성장이 멈춥니다. 스프린트당 1-2개는 의식적으로 🟡 또는 🔴로 도전적이게.

### 6. 시작 값을 비워둔다
Baseline 없이는 진척률을 못 잽니다. 모르면 측정해서 채워넣으세요. "0"으로라도 시작해야 합니다.

### 7. 만들고 안 본다
Project 화면을 주 1회는 열어보세요. 안 보면 죽은 문서가 됩니다.

## 작성 예시

### Task 예시

```markdown
제목: [Task] PoC 모델 정확도 개선

## Tasks 설명
DCASE Task 2 fan 데이터셋 기반의 음향 이상 탐지 PoC가 현재 AUC 0.67 수준에 머물러 있음.
외부 데모와 초기 고객 미팅에 활용하려면 최소 0.85 수준의 신뢰성이 필요.
이번 스프린트는 모델 아키텍처와 특징 추출 파이프라인을 개선하여 목표 AUC를 달성한다.

## Tasks 수행의 목적
현재 AUC로는 외부 PoC 도입처와의 미팅에서 신뢰를 얻기 어려움.
이 Task가 완료되면 데모 가능한 수준의 모델이 확보되어 영업·BD 활동을 본격화할 수 있음.

## Key Results
- [ ] #15 — DCASE fan 데이터셋 AUC 0.85 이상
- [ ] #16 — LSTM-AE 베이스라인 대비 AUC 5%p 이상 개선
- [ ] #17 — 추론 응답시간 200ms 이하 유지

## 예상 결과물
외부 데모용 모델 아티팩트와 평가 리포트가 산출되고,
어떤 입력에 대해서도 정상/이상을 일관되게 판별할 수 있어야 함.

## 책임자
@hayden

## 참고 자료
- 이전 PoC 평가 리포트
- DCASE 2023 Task 2 우승 솔루션 분석
```

### KR 예시

```markdown
제목: [KR] DCASE fan 데이터셋 AUC 0.85 달성

## 해당 Tasks
Parent: #10

## 측정 지표 (Metric)
DCASE Task 2 fan 데이터셋(개발 + 평가셋)에서의 ROC-AUC 점수

## 목표 값 (Target)
AUC ≥ 0.85 (평가셋 기준)

## 시작 값 (Baseline)
AUC 0.67 (이전 PoC, Mel-spec + Autoencoder, 2026-01 측정)

## 측정 방법
- 방법: `eval_auc.py` 스크립트로 평가셋 전체에 대해 추론 후 sklearn `roc_auc_score` 계산
- 데이터/도구: DCASE Task 2 evaluation set, scikit-learn 1.4+
- 측정 주기: 매주 금요일 오후 + 모델 변경 시마다

## 초기 Confidence
→ 🟡 (LSTM-AE 적용 결과에 따라 달라짐)

## 예상 가정·리스크
- LSTM-AE가 Mel-spec AE 대비 충분히 개선되지 않을 가능성
- 평가셋이 학습셋 분포와 너무 달라 일반화 실패 가능성
- 학습 시간이 GPU 자원 제약으로 길어질 가능성

## 관련 작업
- [ ] ml repo #23 — LSTM-AE 베이스라인 구현
- [ ] ml repo #24 — Mel-spec 파라미터 그리드 서치
- [ ] ml repo #25 — 평가 자동화 스크립트
```

## FAQ

**Q. Task와 Objective는 같은 건가요?**<br>
A. 우리 팀에서는 "Task"라고 부르지만, 일반 OKR 용어로는 "Objective"에 해당합니다.<br>
   의식적으로 더 실행적인 단어를 선택했습니다.<br>

**Q. KR을 스프린트 중에 바꿔도 되나요?**<br>
A. 됩니다. 단, 미드 스프린트 리뷰에서 명시적으로 결정하고 변경 이유를 코멘트에 기록해주세요.<br>
   "안 될 것 같아서 살짝 낮춤"은 안 됩니다.<br>
   "외부 데이터 라이선스 문제로 측정 불가, 다른 데이터셋으로 대체" 같은 명확한 이유가 있어야 합니다.<br>

**Q. 1.0 (100% 달성)이 항상 좋은가요?**<br>
A. 아닙니다. **평균 0.7~0.8 정도가 건강합니다.**<br>
   모든 KR을 1.0으로 채운다면 목표가 너무 보수적이었다는 신호입니다.<br>

**Q. Sub-issue를 굳이 코드 repo에 만들어야 하나요?**<br>
A. 네. 코드 작업과 KR이 자동으로 연결되어 "이 KR을 위해 어떤 PR이 머지됐는지" 추적이 자연스러워집니다.<br>
   PR 머지 시 sub-issue도 자동으로 닫혀서 진척률이 자연스럽게 반영됩니다.<br>

**Q. 측정이 정말 어려운 KR은 어떻게 하나요?**<br>
A. 측정이 어렵다면 KR이 아닐 수 있습니다.<br>
   대안:<br>
   (1) 더 구체적인 프록시 지표 찾기,<br> (2) 정성적 평가도 가능한 기준 만들기 (예: "3인 평가단 5점 평균 4.0 이상"),<br> (3) 그래도 안 되면 그건 Task의 sub-issue로 빼고 KR을 다시 잡기.<br>
